## 评审第一轮：方法焦点

### 优势

1. **清晰的问题陈述**：向量搜索与原始数据检索之间的 I/O 协同差距是真实的且研究不足
2. **原则性的机制**：RaBitQ 闭式误差界驱动 I/O 决策，不是启发式阈值
3. **实现成熟度**：530+ 测试、15 个开发阶段、生产级 SIMD 优化
4. **自然的消融结构**：每个组件可独立禁用

### 已解决的弱点

| 弱点 | 解决方案 |
|------|---------|
| 框架过于宽泛 | 缩小到：误差界驱动的 I/O 调度用于 search+payload 协同检索 |
| 贡献过多且平列 | 层级化：SafeOut 剪枝是主力 > 布局、重叠、多阶段为支撑 |
| "技巧组合"感知 | 统一在一个洞察：误差界 → 置信度 → I/O 调度 |
| 无图索引定位 | 分析为什么图索引不适合 payload 协同预取（§5.1） |

## 评审第二轮：初步实验修正

### 初步实验揭示的关键发现

**发现 1：SafeIn ≈ 0%**
- 所有配置（COCO 768维, Deep1M 96维, top_k=10/20）SafeIn 率均 ≈ 0%
- **影响**：三类分类在当前配置下实质上退化为二类（SafeOut / Uncertain）
- **处理**：保留 SafeIn 在框架中，作为理论完备性+探索性贡献（C7），系统性探索激活条件

**发现 2：SafeOut 率随 top_k 递增，CRC 早停率随 top_k 递减**
```
Deep1M:  top_k=10 → SafeOut 93.79%, 早停率 100%, 实际探测 27/100
         top_k=20 → SafeOut 97.91%, 早停率 21%,  实际探测 85.64/100
```
- **影响**：top_k 是一个关键的实验维度，两个机制呈反向关系
- **处理**：在实验中系统性变化 top_k，分析延迟的两面影响

**发现 3：Stage 2 ExRaBitQ 的额外 SafeOut 贡献**
- COCO：Stage 1 SafeOut 90.54% → Stage 2 SafeOut 95.06%（+4.52%）
- Deep1M top_k=10：Stage 1 93.79% → Stage 2 新增 SafeOut 81.95%*
- Deep1M top_k=20：Stage 1 97.91% → Stage 2 新增 SafeOut 88.78%*
- *注：Stage 2 的百分比是对 Stage 1 的 Uncertain 候选的进一步细分
- **影响**：Stage 2 贡献实质但非颠覆性，对 Uncertain 候选的进一步剪枝有价值

## 评审第三轮：评审者可能的疑虑

| 可能问题 | 预先回应 |
|---------|---------|
| "SafeIn 几乎不生效，三类分类名不副实？" | 如实报告；分析 SafeIn 的激活条件空间；论文叙事重心放在 SafeOut 剪枝上，SafeIn 作为潜力探索方向 |
| "为什么不直接与 DiskANN 比纯搜索？" | 我们的贡献在 search+payload 端到端；图索引的指针追踪天然不适合 payload 协同 |
| "CRC 在大 top_k 时失效？" | 如实报告 top_k 敏感性；注意 SafeOut 同时增强，两者呈互补关系 |
| "单线程能代表生产环境吗？" | 聚焦单查询延迟优化；吞吐量通过查询并行线性扩展 |
| "数据集太小/太合成？" | 使用真实 payload 数据集（MS MARCO, COCO 等） |
| "io_uring 比 pread 到底快多少？" | 消融 E2 验证；价值在重叠而非原始 I/O 速度 |

## 最终评审结论

**就绪（附条件）**——方法框架成熟，初步实验提供了真实的数据锚点。SafeIn 的低效率不破坏故事（SafeOut 剪枝才是主要 I/O 节省来源），但需要在论文中诚实处理。CRC 早停的 top_k 敏感性是一个有趣的分析点而非致命缺陷。继续进行实验规划。
