# 实验计划

**问题**：在完成同步暖服务调优和 `.clu` 全量预加载后，BoundFetch 已明显优于 IVF+PQ+FlatStor 系列，但在 COCO 100K 的 `recall@10 >= 0.95` 区间仍落后于调优后的 DiskANN+FlatStor。
**方法论文**：`.clu` 预加载是有效且值得保留的优化，但它本身并不能补齐与 DiskANN 的剩余差距。下一阶段应拆成三条线：公平同步的 IVF 家族基线、面向 BoundFetch 的 CPU 侧定向优化，以及超出 COCO 100K 的分阶段扩展验证。
**日期**：2026-04-15

## 论文故事线
- 主论文必须证明：
  - 暖稳态 E2E 服务仍然是正确的评估目标
  - `.clu` preload 是已经完成且有效的优化，而不是最终答案
  - 下一步有意义的 BoundFetch 优化是 probe / verification 的 CPU 侧效率，而非更多盲目的 `io_uring` 微调
  - 在统一暖协议下，BoundFetch 仍明显强于同步调参后的 IVF 家族基线
  - DiskANN 仍是强图算法上界参考，但不应成为唯一需要正面击败的主轴
- 附录可支撑：
  - 额外的存储后端变体
  - 残余队列深度或模式敏感性检查
  - 图基线参考数据
  - 在 Deep1M / Deep8M 以及一个真实数据负载上的通用性
- 有意裁剪的实验：
  - 冷启动 / drop-cache 协议
  - 完整的 `IVF+PQ/RQ/(DiskANN) x Lance/Parquet/(FlatStor)` 笛卡尔基准测试
  - 在预加载结果明确前又一轮提交路径调优

## 基线对比策略

新基线策略是有意分层的。

- 即使 DiskANN 当前主导暖前沿，也保留它在报告的对比中。
- 不要将 `FlatStor` 作为最终故事中唯一的存储后端。
- 不要尝试完整的搜索后端笛卡尔积，除非自动化使其几乎免费。
- 使用三层对比代替：
  - **层 A：主搜索核心对比**
    - BoundFetch
    - IVF+PQ / IVF+RQ / ConANN 等 IVF 家族方法
    - 若后端策略足够对齐，可附带 DiskANN 概览行
    - 一个选定的存储后端或一个明确记录的后端策略
  - **层 B：存储后端消融**
    - 一到两个代表性搜索核心
    - `FlatStor`、`Lance`，可选 `Parquet`
  - **层 C：构建 / 预加载成本表**
    - 构建时间
    - 索引大小
    - 驻留预加载大小
    - 启动预加载时间

推荐解释：
- 如果对齐的 `Lance` 集成对所有比较系统都可行，优先选择 `Lance` 作为面向生产的主后端。
- 如果对齐后端集成成本太高，保持 `DiskANN+FlatStor` 作为强暖服务上界参考，并将后端格式对比放入层 B。
- 主声明应先在 IVF 家族内部建立，因为这更符合 BoundFetch 的快速构建与未来动态更新定位。
- `Parquet` 只应在工程成本较小时添加；否则归入附录或未来工作。

## 实验块

### 块 1：暖驻聚类预加载
- 测试声明：C1
- 为什么存在此块：这是直接针对当前 DiskANN 差距的最小结构性改变。
- 数据集/分割/任务：`coco_100k`，top-k=10，暖稳态服务，`queries=1000`
- 对比系统：
  - BoundFetch `preload=off`
  - BoundFetch `preload_codes`
  - BoundFetch `preload_codes_and_addr`
- 指标：
  - 主要：`recall@10`、`e2e_ms`、`p99_ms`
  - 次要：`uring_submit_ms`、`probe_ms`、驻留 RSS、预加载时间
- 设置细节：
  - 从当前最佳已知工作点开始：`nlist=2048, nprobe=200, crc-alpha=0.05, epsilon=0.99`
  - 不预加载载荷体
  - 记录驻留内存增长和启动预加载持续时间
- 成功标准：
  - 预加载减少 E2E 延迟或聚类侧时间，足够将 BoundFetch 前沿移向 DiskANN
  - 召回率在相同搜索设置下保持不变
- 失败解释：
  - 如果预加载效果不大，剩余差距可能是搜索质量或算法性的而非热路径元数据访问
- 表格/图形目标：
  - 主论文或附录消融表
  - 分解图显示预加载后什么移动了
- 优先级：必须运行

### 块 2：预加载后刷新的 BoundFetch 帕累托
- 测试声明：C1
- 为什么存在此块：一次预加载消融不够；需要看整个 BoundFetch 前沿是否移动。
- 数据集/分割/任务：`coco_100k`，top-k=10，暖稳态服务，`queries=1000`
- 对比系统：
  - BoundFetch 当前前沿
  - BoundFetch 启用预加载
- 指标：
  - `recall@10`
  - `e2e_ms`
  - `p99_ms`
- 设置细节：
  - 在 `crc-alpha={0.01,0.02,0.05,0.08,0.1}` 上重新运行紧凑网格，`nprobe={200,500}`
  - 只有在预加载实质性改变前沿形状时才添加 `nlist` 或 `epsilon` 细化
- 成功标准：
  - 至少出现一个新的非支配 BoundFetch 点
- 失败解释：
  - 如果刷新的前沿仍完全被 DiskANN 支配，停止低级调优并将论文重新定位到构建时间/集成权衡
- 表格/图形目标：
  - 更新的主帕累托图
- 优先级：必须运行

### 块 3：BoundFetch 的 CPU 侧优化
- 测试声明：C2
- 为什么存在此块：在 preload 完成后，剩余差距最可能来自 probe / verification 的 CPU 成本，尤其是在 `recall@10 = 0.95` 以上区间。
- 数据集/分割/任务：`coco_100k`，top-k=10，暖稳态服务，`queries=1000`
- 对比系统：
  - BoundFetch 当前最佳 preload 设置
  - 每次只引入一个 CPU 路径优化后的 BoundFetch
- 指标：
  - `recall@10`
  - `e2e_ms`
  - `p99_ms`
  - `probe_ms`
  - `rerank_cpu_ms`
  - 扫描/验证候选数
- 设置细节：
  - 固定 `clu_mode=full_preload`
  - 以两个高价值工作点为锚点：
    - `nlist=2048, nprobe=200, alpha=0.02`
    - `nlist=2048, nprobe=200, alpha=0.01`
  - 仅实现那些能在不改变搜索语义下减少 per-cluster 或 per-candidate CPU 开销的改动
  - 候选方向包括：
    - 降低重复 decode / bound-check 工作
    - 优化 probe 阶段的数据布局和 SIMD 友好性
    - 缩减 candidate materialization 与 rerank 代价
    - 将 payload 请求准备与 probe 计算进一步重叠
- 成功标准：
  - 至少有一个 `recall@10 >= 0.95` 的工作点在不改变召回语义下明显向 DiskANN 靠近
- 失败解释：
  - 如果 CPU 侧调优无法实质推动 `>=0.95` 区间，就停止继续追逐 DiskANN 的暖延迟
- 表格/图形目标：
  - 机制表 + 刷新的高召回 Pareto
- 优先级：必须运行

### 块 4：同步 IVF 家族基线
- 测试声明：C3
- 为什么存在此块：在继续追 DiskANN 之前，需要先证明 BoundFetch 在它最自然的同类方法中处于什么位置。
- 数据集/分割/任务：`coco_100k`，代表性暖服务工作点
- 对比系统：
  - BoundFetch 最佳 preload 点
  - FAISS IVF+PQ
  - 若可用，再补一个 IVF 家族基线，优先 IVF+RQ 或 ConANN
- 指标：
  - `recall@10`
  - `e2e_ms`
  - `p99_ms`
  - 构建时间
- 设置细节：
  - 在相同暖协议下，将每个 baseline 都调成自己的 Pareto 曲线
  - 尽量统一 `queries=1000`、`topk=10`，以及可比时统一 `nlist`
  - 形成一张同步图，至少包含：
    - BoundFetch
    - IVF+PQ
    - IVF+RQ / ConANN
- 成功标准：
  - 建立 BoundFetch 在 IVF 家族 tradeoff surface 上的领先位置
- 失败解释：
  - 如果另一个 IVF 家族方法支配 BoundFetch，应先解释并修复这一差距，再考虑扩数据集
- 表格/图形目标：
  - 主 Pareto 图中的 IVF 家族对比
- 优先级：必须运行

### 块 5：DiskANN 参考与成本表
- 测试声明：C3
- 为什么存在此块：DiskANN 应该继续保留，但更适合作为强图算法参考，再配合构建/启动成本形成完整权衡。
- 数据集/分割/任务：`coco_100k`
- 对比系统：
  - BoundFetch
  - DiskANN
  - 一个调优后的 IVF 家族基线
- 指标：
  - `recall@10`
  - `e2e_ms`
  - `p99_ms`
  - 构建时间
  - 构建期间峰值 RSS
  - 索引字节
  - 预加载字节
  - 预加载时间
- 设置细节：
  - 将这些指标与查询延迟分开报告
  - 将构建和预加载成本作为一级证据，而非脚注
- 成功标准：
  - 产生一张清晰表格，显示即使 DiskANN 在服务上更快，BoundFetch 是否仍占据更好的构建时间/启动/简单性点
- 失败解释：
  - 如果 BoundFetch 在构建和启动成本上也输了，论文范围必须再次缩小
- 表格/图形目标：
  - 主成本表 + 次级 DiskANN 参考图
- 优先级：必须运行

## 计算与数据预算
- 预计 GPU 小时：0
- 数据准备需求：
  - 无需新的 COCO 准备
  - 可选的 `Lance` 或 `Parquet` 后端转换工作
- 人工评估需求：无
- 最大瓶颈：工程时间，而非计算
- 最大协调规则：保持实验套件分层；不让搜索后端矩阵爆炸
- preload 之后的直接瓶颈：高召回区间的 CPU 搜索成本

## 风险与缓解措施
- 风险：CPU 侧调优也无法足够推动高召回区间。
  - 缓解：停止继续正面追击 DiskANN 的暖延迟，将主声明转到 IVF 家族领先性与构建/启动优势。
- 风险：如果 DiskANN 使用 FlatStor 而其他系统使用另一后端，评审人质疑公平性。
  - 缓解：要么在层 A 中对齐后端，要么明确标记 DiskANN+FlatStor 为上界参考，并分开展示后端消融。
- 风险：基线空间变得太大无法干净执行。
  - 缓解：冻结分层基线策略并拒绝完整笛卡尔扩展。
- 风险：`Parquet` 成为工程时间的消耗而无助于改变结论。
  - 缓解：只在 `FlatStor` vs `Lance` 稳定后才添加 `Parquet`。
- 风险：真实数据集引入 schema 和 payload 异质性，拖慢主故事收敛。
  - 缓解：等 COCO + Deep 的流程稳定后再上真实数据。
