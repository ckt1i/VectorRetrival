# Research Proposal: BoundFetch-Single 本科论文最小完整实验方案

## Problem Anchor

- Bottom-line problem: 在本科毕业论文时间约束下，将当前 VectorRetrival/BoundFetch 代码收敛为一版最小但完整、可复现、可写作的实验闭环，证明集成式向量检索与 payload 读取协同设计在真实检索和合成 payload 场景中优于拆分式 baseline。
- Must-solve bottleneck: 旧方向同时展开 CRC 早停、RAIR/重复聚类、CPU 热路径优化和大 baseline 矩阵，实验面过宽；当前必须缩小到单聚类划分、无 CRC 早停、有限数据集、有限 baseline 的主实验与 ablation，使论文能按期完成。
- Non-goals: 不证明 ConANN CRC 早停；不做 RAIR 或重复聚类；不做完整 `search x storage x dataset` 笛卡尔矩阵；不追求击败所有图索引或商业向量数据库；不新增复杂学习模块。
- Constraints: 数据集只考虑 `coco_100k`、`deep1m_synth`、`msmarco_passage`；其中 `deep1m_synth` 仍需生成，当前 `/home/zcq/VDB/data/formal_baselines` 下看到的是 `deep8m_synth`；baseline 只考虑 `IVF+PQ/IVF+RaBitQ` 与 `FlatStor/Lance` 的四种组合；当前代码执行时固定 `--crc 0`、`--assignment-mode single`，不启用 RAIR/重复聚类；复用 `/home/zcq/VDB/baselines` 现有脚本和结果格式；论文写作参考 VLDB/PVLDB 的系统论文结构。
- Success condition: 产出一套可执行实验矩阵、主实验图表、关键 ablation、方法叙事、创新点和毕业论文大纲；即使最终只完成 COCO 100K 与一个扩展数据集，也能形成自洽论文。

## Final Thesis

本文提出 `BoundFetch-Single`：一个 single-assignment IVF 下的 integrated `retrieve -> fetch -> rerank` query plan。它固定 top-`C` 候选预算，将 RaBitQ 近似候选生成、payload/原向量读取和 exact rerank 放进同一个服务路径，并在 matched-recall 条件下比较完整 E2E latency。

## 方法边界

- 主方法只研究 single-assignment IVF。
- 默认关闭 CRC：`--crc 0`。
- 不做 RAIR、duplicate clustering、learned probe policy 或额外 reranker。
- `Lance` 与 `msmarco_passage` 只作为 appendix/backlog，不进入主线成功条件。

## BoundFetch-Single Query Protocol

每个 query 固定以下步骤：

1. 对 centroids 打分，选择 `nprobe` 个 cluster。
2. 在 probed clusters 内用 RaBitQ approximate scorer 产生候选。
3. 选取 top-`C` 候选，`C in {64,128,256}`；如果当前 runner 没有显式 `C`，则使用等价 rerank/candidate budget，并在 run log 中记录映射。
4. top-`C` 候选确定后提交原向量/payload 读取。
5. 在 top-`C` 上做 exact rerank。
6. 返回 top-`k` payload-bearing results，默认 `topk=10`。

公平性规则：

- 主表采用 matched-recall 口径：对所有系统取共同可达 recall target `R*`，`R*` 是所有主线系统都能达到的最高共同 recall 区间；每个系统报告达到 `R*` 的最低 E2E latency。
- 主线 baseline 固定为 `IVF+PQ+FlatStor` 和 `IVF+RaBitQ+FlatStor`。
- 主表比较完整 `retrieve + fetch + rerank` E2E latency，不用单独 ANN latency 代替。

## 主实验方案

### Mandatory: COCO100K 主闭环

- Dataset: `coco_100k`
- Systems: `BoundFetch-Single`、`IVF+PQ+FlatStor`、`IVF+RaBitQ+FlatStor`
- Fixed: `topk=10`、`queries=1000`、`nlist=2048`、`--crc 0`、`--assignment-mode single`
- Sweep: `nprobe={64,128,256}`；只有主图需要更完整曲线时补 `512`
- Figures/Tables:
  - 主表：matched `recall@10` 下 `avg/p95/p99 E2E latency`
  - 主图：COCO recall-latency curve
  - 分解图：ANN / fetch / wait / rerank latency

### Required Ablation: overlap on/off

- Dataset: `coco_100k`
- Systems: `BoundFetch-Single overlap on` vs `serial fetch / overlap off`
- Clean ablation rule: 除关闭 overlap 外，不改变 candidate budget、cache protocol、submission mode、nprobe、topk 或 payload backend。
- Metrics: `avg/p95/p99 E2E latency`、wait time、fetch latency、rerank latency。

### Second: deep1m_synth payload sensitivity

- Dataset: `deep1m_synth`
- Mandatory payload size: `4KB`
- Optional payload sizes: `256B` and `16KB/64KB`
- Systems: `BoundFetch-Single`、`IVF+PQ+FlatStor`、`IVF+RaBitQ+FlatStor`
- Fallback: 如果 `deep1m_synth` 生成拖期，用 COCO 记录加 synthetic payload padding 做 sensitivity，不阻塞主线论文。

### Appendix / Backup

- `Lance`: 只有 runner 已经稳定复用时补 `IVF+PQ+Lance` 和 `IVF+RaBitQ+Lance`。
- `msmarco_passage`: 只有 embedding、query、qrels/ground truth 都可用时作为文本 workload sanity check。

## Warm-Serving / Cache Protocol

- 每个系统使用同一批 query id。
- 每个配置先 warmup，再正式计时；首轮结果不进主表。
- baseline 与 BoundFetch 使用相同 cache 口径：同为 warm cache 或同为显式 drop-cache，不混用。
- 记录 binary path、git commit、dataset path、nprobe、candidate budget、cache setting、run timestamp。
- 主报告优先使用 `p95/p99` 和 matched-recall E2E latency。

## 可写进论文的创新点

1. Integrated retrieve-fetch-rerank query plan: 用固定 top-`C` 候选预算定义读取和 rerank 边界，避免 ANN 与 payload fetch 完全分离。
2. Minimal single-assignment system design: 保留 IVF 构建和解释简单性，证明不依赖 CRC/RAIR 也能形成端到端系统收益。
3. Matched-recall E2E evaluation: 以对象返回服务链路为评估对象，报告 avg 与 tail latency，而不是只报告 ANN kernel 时间。
