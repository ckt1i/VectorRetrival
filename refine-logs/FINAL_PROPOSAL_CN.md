# 最终提案：基于误差界的磁盘向量搜索与原始数据协同优化

**日期**：2026-04-10（v3: 基于 top_k=10/20 初步实验全面修正）
**状态**：就绪
**目标会议**：SIGMOD / VLDB / OSDI（系统类+数据类）

---

## 1. 问题陈述

在生产级向量搜索系统（RAG、多模态检索、推荐系统）中，查询不仅要返回最近的向量，还要返回关联的原始数据（文档、元数据、图片）。在基于磁盘的系统中，**I/O 占查询延迟的 70–90%**（PageANN 2024, PipeANN OSDI 2025）。

现有磁盘 ANNS 系统（DiskANN、SPANN、Starling）将向量索引遍历和原始数据检索视为**独立的、顺序的两个阶段**：
- **方案 A（先搜再取）**：检索所有候选向量，计算精确距离，然后读取 Top-K 的原始数据。问题：原始数据读取发生在搜索完成后，增加完整的 I/O 往返时间到尾延迟。
- **方案 B（贪心预取）**：共存向量和原始数据，获取所有数据。问题：浪费带宽在不会进入 Top-K 的候选上（通常 >90% 的获取数据被丢弃）。

**关键空白**：现有系统没有利用近似距离估算的*置信度*来做细粒度的、逐候选的 I/O 调度决策——跳过高置信度的非 Top-K 候选、提前预取高置信度的 Top-K 候选、仅为不确定的候选做精确验证和原始数据获取，所有操作都与搜索计算本身重叠。

---

## 2. 方法论述（一句话）

我们利用 RaBitQ 的可控量化误差界作为置信信号，对每个候选进行三类分类（SafeIn / SafeOut / Uncertain），通过 SafeOut 剪枝 90%+ 的候选避免无效 I/O，通过 SafeIn 对高置信候选提前预取完整记录，仅对少量 Uncertain 候选执行精确验证和按需原始数据获取——全程通过 io_uring 异步执行，与聚类探测并行。

---

## 3. 主要贡献

**误差界驱动的三类 I/O 调度**：一个统一框架，连接近似距离估算与 I/O 资源分配。关键洞察是 RaBitQ 的理论误差界 `epsilon = c * 2^{-bits/2} / sqrt(dim)` 提供了**逐候选、逐聚类的 margin**，转换为三个 I/O 操作：

- **SafeOut** (`approx_dist > d_k + 2 * margin`)：候选**保证不**在 Top-K 中 → 跳过（零 I/O）。初步实验显示 **Stage 1 剪枝 90-98% 的候选**
- **Uncertain** (介于两阈值之间)：无法确定 → 获取向量，计算精确距离，若进入 Top-K 再获取原始数据
- **SafeIn** (`approx_dist < d_k - 2 * margin`)：候选**保证**在 Top-K 中 → 可直接预取完整记录（向量+原始数据），跳过精确距离验证

### SafeIn 的现状与潜力

初步实验中，SafeIn 率在当前配置下接近 0%。这是因为 SafeIn 条件要求 `approx_dist` 远小于动态 `d_k` 减去 margin，在以下因素叠加下条件极为严格：
1. CRC 校准的 `d_k` 取 P99 分位数（偏保守）
2. 1-bit FastScan 的误差界较宽
3. top_k 较小时堆较满，`d_k` 较紧

**SafeIn 可能在以下场景显现价值**（留作实验验证）：
- **更大 top_k**（如 top_k=50/100）：堆阈值 d_k 更宽松
- **更高量化精度**（8-bit ExRaBitQ）：margin 更小，SafeIn 条件更易满足
- **特定数据分布**：聚类内方差小、距离分布集中的数据集
- **更激进的 d_k 校准**：使用 P90 而非 P99 分位数
- **大 payload 场景**：SafeIn 可完全消除 Uncertain 的"先读向量再读 payload"两轮 I/O

> **定位**：SafeIn 在三类框架中具有理论完备性和系统设计完备性（代码已实现），在当前实验配置下效果有限，但作为一个随条件变化可能显现价值的机制，我们将其保留在框架中并在实验中系统性探索其激活条件。

## 4. 支撑贡献

### 4.1 协同设计的双文件存储布局

两个互补的存储文件，为流水线的两个阶段优化：

- **ClusterStore (.clu)**：量化代码（FastScan 打包）+ 地址列（比特打包的记录位置）按 IVF 聚类连续存储。支持顺序扫描，最小化 I/O 放大。
- **DataFile (.dat)**：原始向量与原始数据列按记录共存，4KB 页对齐。支持单次读取检索完整记录。DataFile 的设计参考了 FlatStor（VLDB 2025）的 FlatBuffers 嵌入式索引思路。

**地址列**使用 SIMD 加速的比特解包（K=8 多流并行前缀和）来批量解码 64 个记录位置，桥接聚类级扫描到记录级随机 I/O。

> **与 FlatStor 的关系**：FlatStor（VLDB 2025）优化 "给定 ID 集合，高效读 payload"（被动响应），是多模态 payload 随机访问的 SOTA。BoundFetch 解决更上层问题："在搜索过程中，利用误差界决定哪些 payload 值得读，并与搜索并行读取"（主动调度）。两者是不同层次的优化——BoundFetch 的 DataFile 可替换为 FlatStor 格式而不影响上层 I/O 调度框架。

### 4.2 带滑动窗口预取的重叠调度（单线程模型）

一个基于 io_uring 的**单线程**调度器，维护三个并发流水线：
1. **聚类块预取**：即将探测的聚类块的滑动窗口（可配置深度 4–16）
2. **候选 I/O 分发**：分类驱动的读操作，在识别候选时提交
3. **重排序消费**：消费完成的 I/O 缓冲区，进行精确距离计算和原始数据缓存

调度器使用统一的事件循环，非阻塞 CQE 轮询，交织 CPU 绑定探测与 I/O 完成处理。

> **设计范围**：当前实现为单线程、单 io_uring ring 模型，聚焦于单查询延迟优化。查询级吞吐量可通过多线程（每线程独立处理一个查询）线性扩展，这是正交的系统工程问题，不影响 I/O 调度框架的有效性。

### 4.3 多阶段量化流水线

- **阶段 1（FastScan）**：AVX-512 VPSHUFB 批处理-32，14 比特量化查询 vs 1 比特代码。提供粗粒度距离估计，**已实现 90-98% SafeOut 率**（随 top_k 变化）。
- **阶段 2（ExRaBitQ）**：完整 M 比特（2/4 比特）估计仅应用于阶段 1 的 Uncertain 候选。进一步紧缩误差界。
- **动态阈值（CRC）**：维护当前 Top-K 距离的运行估计堆，随搜索进行逐步紧缩 SafeOut 阈值。

## 5. 故意舍弃的复杂性

- **图索引（DiskANN/Vamana）**：图遍历是指针追踪式 I/O，每步依赖上步结果，无法提前预测下一个节点，因此无法做批量 I/O 调度或基于误差界的候选剪枝。IVF 的批量顺序聚类扫描天然适合此框架。详见§5.1。
- **多线程并发**：当前聚焦单线程单查询延迟。多线程引入锁竞争、io_uring ring 共享等复杂性，是正交的工程问题。吞吐量可通过查询级并行线性扩展。
- **学习索引 / ML 路由**：增加训练开销和模型服务复杂性，相比 RaBitQ 的闭式误差界无明确益处。
- **分布式 / 多节点设计**：限定于单节点磁盘系统。可按 IVF 聚类自然分片。
- **GPU 加速**：瓶颈是磁盘 I/O，不是计算。
- **过滤谓词（filtered search）**：正交关注，留作未来工作。

### 5.1 为什么图索引不适合 Payload 协同预取？

| 特性 | IVF (BoundFetch) | 图索引 (DiskANN) |
|------|-----------------|------------------|
| 遍历模式 | 批量扫描一个聚类内所有候选 | 逐节点指针追踪 |
| I/O 可预测性 | 高：提前知道要读哪些聚类块 | 低：下一个节点取决于当前节点的邻居 |
| 批量 I/O 调度 | 可以：一次对整个聚类做分类决策 | 不可以：每次只处理一个节点 |
| 误差界利用 | RaBitQ 提供全局误差界用于 SafeOut/SafeIn | 无类似机制 |
| Payload 预取时机 | 可在聚类探测时并行提交 | 必须等图搜索完成后顺序获取 |
| 索引构建时间 | 快（K-means） | 慢（图构建 + 边剪枝） |

**结论**：图索引在纯向量搜索中确实优于 IVF，但在 search+payload 协同检索场景中，IVF 的批量可调度性使其成为更好的 I/O 调度基座。我们不与图索引在纯向量搜索上竞争，而是在 **search+payload 端到端** 上展示优势。

## 7. 必须进行的消融

1. **分类影响**：完整三类 vs 仅 SafeOut（无 SafeIn）vs 无分类（所有候选读向量）vs 贪心（全读完整记录）
2. **SafeIn 激活条件探索**：变化 top_k, bits, d_k 校准分位数，系统性报告 SafeIn 率
3. **阶段 2 必要性**：仅 1 比特 vs 1+4 比特
4. **重叠调度**：io_uring 重叠 vs 同步 I/O（pread）
5. **存储布局**：共存 vs 分离
6. **CRC 早停**：有 vs 无，以及 top_k 对早停率的影响
7. **预取深度**：滑动窗口大小敏感性（1, 4, 8, 16）
8. **原始数据大小敏感性**：0, 1KB, 10KB, 100KB, 1MB

## 9. 论文定位

**标题**："BoundFetch：基于量化误差界的磁盘向量搜索 I/O 调度与原始数据协同检索"

**单段摘要**：磁盘 ANNS 系统必须同时交付最近的向量和关联的原始数据。现有系统要么在搜索后顺序获取原始数据（增加一轮 I/O 延迟），要么贪心预取所有候选（浪费 90%+ 的 I/O 带宽）。我们提出 BoundFetch，利用 RaBitQ 量化的理论误差界对候选进行三类 I/O 调度——SafeOut 剪枝（实验表明单阶段即消除 90-98% 候选）、SafeIn 预取（对高置信度 Top-K 候选直接获取完整记录）和 Uncertain 按需验证。结合 CRC 动态阈值、协同设计的双文件存储布局和基于 io_uring 的单线程重叠调度器，BoundFetch 在 search+payload 端到端指标上相比现有磁盘 ANNS 系统实现显著更低延迟，同时将原始数据交付融入搜索流程。

## 10. 初步实验数据锚点

### COCO 100K（768 维，nlist=2048，nprobe=150，top_k=10，bits=4，CRC on）
```
recall@10 = 0.8925, latency avg = 1.738ms, p99 = 2.084ms
avg_probed = 102/150, early_stop_rate = 100%
Stage 1: SafeOut 90.54%, Uncertain 9.46%, SafeIn 0%
Stage 2: SafeOut 95.06%, Uncertain 4.94%, SafeIn 0%
```

### Deep1M top_k=10（96→128 dim，nlist=4096，nprobe=100，bits=4，CRC on）
```
recall@10 = 0.9230, latency avg = 0.814ms, p99 = 1.376ms
avg_probed = 27/100, early_stop_rate = 100%
Stage 1: SafeOut 93.79%, Uncertain 6.21%, SafeIn 0%
Stage 2: SafeOut 81.95%, Uncertain 17.99%, SafeIn 0.05%
```

### Deep1M top_k=20（96→128 dim，nlist=4096，nprobe=100，bits=4，CRC on）
```
recall@10 = 0.9630, latency avg = 1.547ms, p99 = 2.001ms
avg_probed = 85.64/100, early_stop_rate = 21%
Stage 1: SafeOut 97.91%, Uncertain 2.09%, SafeIn 0%
Stage 2: SafeOut 88.78%, Uncertain 11.19%, SafeIn 0.03%
```

### top_k 敏感性初步观察（Deep1M）

| 指标 | top_k=10 | top_k=20 | 趋势解读 |
|------|----------|----------|---------|
| Stage 1 SafeOut | 93.79% | 97.91% | top_k↑ → d_k 更宽松 → SafeOut 更激进 |
| Stage 1 Uncertain | 6.21% | 2.09% | Uncertain 减少 |
| CRC 早停率 | 100% | 21% | top_k↑ → 堆更难填满 → 早停效果下降 |
| 实际探测聚类 | 27/100 | 85.64/100 | 早停失效导致更多聚类被探测 |
| latency avg | 0.814ms | 1.547ms | +90%，主要因早停失效 |
| SafeIn | 0% | 0.03% | 微有改善，仍 ≈ 0 |

**关键发现**：SafeOut 剪枝率与 CRC 早停率呈**反向关系**：top_k 小时早停效果强（73% 节省），top_k 大时 SafeOut 率更高（98%）但早停弱。这是一个有趣的权衡，可在论文中深入分析。
