# 最终提案：暖稳态集成向量搜索与载荷协同检索

**日期**：2026-04-13  
**状态**：就绪  
**目标会议**：SIGMOD / VLDB / 系统-数据交叉方向

## 问题锚点
- 核心问题：在生产级向量检索场景中，查询路径应被评估为 `搜索 + 载荷传递` 的端到端整体，而非单独评估向量搜索。
- 必须解决的瓶颈：现有的分离式流水线要么在搜索完成后承受第二阶段载荷检索的额外开销，要么对最终未进入 Top-K 的候选项做了无用功。在当前部署环境下，剩余的延迟瓶颈已不再是块设备等待，而是集成探针、提交与候选项验证的 CPU 路径。
- 非目标：本项目不针对冷启动延迟、严格的 drop-cache 协议或首次查询缺页行为。在不含载荷传递的情况下，也不试图在纯 ANN 延迟上超过图索引。
- 约束：服务器不提供 `sudo`；暖稳态是唯一必须支持的协议。代码库应保持单节点、低复杂度，聚焦于一个主要机制贡献，而非多个并列贡献。
- 成功标准：在暖稳态服务下，展示 BoundFetch 相对于强分离基线在端到端召回率-延迟权衡上具有更优或更有利的结果，并通过紧凑的机制层面分析解释其增益来源。

## 技术差距
早期提案过度强调冷 I/O 和宽泛的三类调度故事。当前的实验证据指向一个更清晰的方向：

1. 有意义的场景是暖稳态服务。
2. `io_wait` 已几乎完全被隐藏；实际瓶颈是提交侧 CPU 加上固定的探针侧开销。
3. 最强的竞争对比不是"谁在冷启动上最好"，而是"在生产级稳态下，谁能给出最好的端到端搜索+载荷结果"。
4. SafeIn 在结构上有趣，但目前不是核心证据；SafeOut 以及集成提交/路径设计才是更强的主导故事。

因此，论文不应扩展为通用磁盘 I/O 论文，而应聚焦为一篇关于暖服务条件下集成搜索+载荷检索的系统论文。

## 方法核心论点
- 一句话论点：BoundFetch 在暖稳态下胜出，是因为它将候选项过滤、载荷访问和异步提交集成到一条检索路径中，相比分离式"先搜索后获取"基线减少了端到端工作量。
- 为何这是最小充分干预：胜出机制已存在于代码库中；有意义的下一步不是新增子系统，而是在大致相同的延迟下提升召回率，并用更清晰的证据验证该机制。
- 为何此方向适时：生产级向量系统越来越关注完整的服务延迟，而非仅仅搜索延迟，尤其是当载荷传递和元数据访问被纳入评估时。

## 贡献聚焦
- 主要贡献：一条暖服务集成检索路径，将向量探针与载荷协同检索整合为一个端到端系统，在 E2E 延迟上优于强分离基线。
- 可选辅助贡献：提交路径优化与机制归因，解释剩余延迟的去向，以及为何额外的队列深度或 SQPOLL 调优不是主要的下一个优化方向。
- 明确的非贡献：
  - 冷启动优越性
  - 纯 ANN 场景的图搜索霸主地位
  - 大规模内核或 OS 调优声明
  - 将 SafeIn 作为增益的主要来源

## 提议方法
### 复杂度预算
- 冻结/复用的基础组件：
  - 现有的基于 IVF 的 BoundFetch 检索框架
  - 现有的双文件/集成存储格式
  - 现有的暖基准测试框架和基线流水线
- 新的可训练组件：无
- 明确避免引入的新系统组件：
  - 新的索引族
  - 新的并发模型
  - GPU / 分布式 / 学习型控制器

### 系统概览
1. 探测 IVF 聚类并计算候选项得分。
2. 使用当前的过滤路径抑制明显不必要的候选项工作。
3. 通过优化后的 `io_uring` 路径提交载荷/向量读请求。
4. 在同一集成流水线内完成解析、重排序和 Top-K 传递。
5. 在暖稳态下将整个系统作为一条端到端服务路径进行测量。

### 核心机制
- 输入/输出：
  - 输入：查询向量、索引状态、载荷支撑数据文件
  - 输出：Top-K 检索结果加上载荷访问，在一次 E2E 延迟测量中完成
- 架构或策略：
  - 集成探针 + 异步提交 + 按需验证
  - 共享提交模式作为默认的暖服务路径
- 训练信号/损失：不适用；这是系统机制，非学习模型
- 为何这是主要创新点：
  - 贡献不仅仅是"使用 io_uring"
  - 贡献在于：搜索路径与载荷路径不再作为独立阶段进行评估或执行

### 可选辅助组件
- 组件：提交路径优化与批处理
- 作用：降低每个查询的提交开销，使集成检索在低毫秒级延迟下保持竞争力
- 为何不会造成贡献分散：这是对主导集成路径声明的证据支撑和工程支持，而非第二篇独立论文

### 现代原语使用
- 前沿原语使用：无
- 原因：本工作不需要强行引入 LLM/VLM/RL 组件；最强的故事是一个有意简洁的系统贡献

### 下游流水线集成
- BoundFetch 应被定位为服务路径系统，而非独立的 ANN 核心。
- 评估目标是返回可用结果的应用可见延迟，包括载荷访问。
- 因此，基线对比必须保持端到端且仅限暖态协议。

### 面向执行的计划
- 保留当前优化后的提交路径。
- 仅在能明确提升召回率且延迟基本不变的情况下才追加优化预算。
- 优先进行参数级和阈值级调优，而非结构性重设计。

### 故障模式与诊断
- 故障模式：BoundFetch 虽然比 DiskANN+FlatStor 更快，但召回率显著低于目标范围。
  - 通过召回率-延迟帕累托曲线检测。
  - 缓解措施：调整 `CRC_alpha`、裁剪阈值以及轻量级重排序/获取策略。
- 故障模式：进一步的提交路径工作收益可忽略。
  - 通过 `uring_submit_ms`、`submit_calls` 和 E2E 延迟无变化来检测。
  - 缓解措施：停止系统微优化，将精力转移到召回率提升和证据收紧。
- 故障模式：评审人将增益解读为召回率权衡而非真实的系统胜出。
  - 通过缺少近等召回率比较点来检测。
  - 缓解措施：运行紧凑的召回率改进消融实验，以帕累托曲线代替单点报告。

### 创新性与简洁性论据
论文的最强版本不是"我们为磁盘 ANN 构建了许多组件"，而是：

"一旦服务目标被正确定义为暖稳态 `搜索 + 载荷`，集成检索路径就能在端到端维度上击败分离基线，且剩余瓶颈是提交/探针 CPU 而非设备 I/O。"

这比更宽泛的冷 I/O 或通用 ANN 声明更紧凑、更具可辩护性。

## 声明驱动的验证概要
### 声明 C1：BoundFetch 在强分离基线上提供更优的暖态 E2E 帕累托
- 最小实验：仅使用暖协议在 COCO 100K 上进行端到端对比，BoundFetch sweep 对比 DiskANN+FlatStor 和 FAISS-disk+FlatStor。
- 基线/消融：
  - BoundFetch `nprobe` sweep
  - 已有的最强分离基线点
- 指标：
  - `recall@10`
  - `e2e_ms`
  - `p99_ms`
- 预期证据：
  - BoundFetch 在至少一个实际相关的工作区间内占优
  - BoundFetch 在延迟上明显优于 DiskANN+FlatStor
  - FAISS-IVFPQ-disk 在 COCO 上的召回率受限被明确记录

### 声明 C2：剩余的工程价值在于提升召回率，而非更多的提交路径微优化
- 最小实验：在固定暖协议下，对 `CRC_alpha`/裁剪策略/可选重排序行为进行紧凑消融。
- 基线/消融：
  - 当前最优配置
  - 2-3 个相邻阈值设置
  - 可选的关闭裁剪或放宽阈值变体
- 指标：
  - 在可比延迟下的召回率提升
  - `uring_submit_ms`、`submit_calls`、`probe_ms`
- 预期证据：
  - 召回率可以在 E2E 延迟基本不变的情况下有意义地移动
  - 队列深度/SQPOLL/提交模式 sweep 仍属次级验证，不是主要未来工作

## 实验交接输入
- 必须证明的声明：
  - BoundFetch 应作为暖服务集成端到端系统进行评估
  - 论文的下一步收益应来自更好的召回率-延迟权衡，而非更低的原始提交开销
- 必须运行的消融：
  - 召回率-延迟帕累托 sweep

  - 当前召回率天花板可能仍然太低，无法形成有说服力的主论文对比
