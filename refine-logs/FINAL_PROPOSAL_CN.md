# 最终提案：暖驻 BoundFetch 与分层基线评估

**日期**：2026-04-14
**状态**：同步基线调优后更新
**目标会议**：SIGMOD / VLDB / 系统-数据交叉方向

## 问题锚点
- 核心问题：在生产级向量检索中，查询路径应被评估为 `搜索 + 载荷传递`，而非孤立的向量搜索。
- 必须解决的瓶颈：在同步暖服务调优后，BoundFetch 明显强于 IVF+PQ+FlatStor 系列，但在 COCO 100K 上仍被调优后的 DiskANN+FlatStor 主导。下一步是确定最小暖驻预加载 `.clu` 量化向量和原始地址元数据是否可以弥补足够的服务差距，以恢复有用的系统位置。
- 非目标：
  - 冷启动延迟或 drop-cache 协议
  - 仅因为 DiskANN 当前更强就将其从对比中移除
  - 在完整的笛卡尔网格中基准测试每个 `search x storage` 组合
  - 将完整载荷体预加载到内存然后称结果为磁盘服务系统
- 约束：
  - 服务器不提供 `sudo`
  - 暖稳态是唯一必须的服务协议
  - 代码库应保持单节点且复杂度适中
  - 载荷数据应保持在正常服务路径上；只有轻量级聚类侧元数据可以提升到内存
- 成功条件：要么展示带暖驻聚类元数据的 BoundFetch 达到相对于 DiskANN 的有用非支配区域，同时仍明显强于 IVF 家族基线；要么展示明确的构建时间/简单性/服务权衡，其中 DiskANN 保持搜索上界但非全轴系统赢家。

## 技术差距
最新的同步基线结果以一种重要方式改变了故事：

1. BoundFetch 在暖 E2E 延迟上简单击败强分离基线这一旧论点不再准确。
2. 当前 BoundFetch 前沿明显高于 FAISS-IVFPQ+FlatStor，但调优后的 DiskANN+FlatStor 现在定义了最强的暖前沿。
3. 剩余差距不太可能仅通过更多队列深度或提交路径微调来弥补；当前证据指向 `.clu` 路径中的结构性每查询工作。
4. 搜索核心效应和存储格式效应现在纠缠在一起。如果我们将 `BoundFetch + FlatStor` 与 `DiskANN + FlatStor` 对比，我们仍然不知道有多少来自搜索策略与后端格式。

因此，下一个论文级行动不是扩大系统，而是对 BoundFetch 进行一个小的结构性改进，并重新设计基线逻辑以保持强对比的公平性和可解释性。

## 方法论文
- 一句话论文：最小下一步干预是在查询执行前预加载 `.clu` 量化向量和原始地址块到暖驻内存，使 BoundFetch 能够从热路径中移除重复的聚类侧读/解析工作，而不预加载载荷体。
- 为什么这是最小充分干预：它直接针对当前相对于 DiskANN 的结构劣势，DiskANN 已受益于将其压缩搜索侧表示驻留在内存中。
- 为什么评估计划也必须改变：基线故事应分离两个轴——搜索核心质量和存储后端质量——而不是扩展成既昂贵又难解释的大型矩阵。

## 贡献聚焦
- 主要贡献：一条暖驻 BoundFetch 服务路径，在查询前缓存聚类侧量化向量和地址元数据，同时保持载荷访问在正常集成路径上。
- 可选辅助贡献：一种分层基线方法论，保持 DiskANN 作为强必需参考，保持 IVF+PQ/RQ 作为较低搜索基线，并将存储格式效应隔离在单独的对比块中。
- 明确非贡献：
  - 声称纯 ANN 优越于调优后的图索引
  - 声称 FlatStor 是唯一有意义的存储后端
  - 声称每个搜索族必须与每个存储后端在主论文中配对

## 提议方法
### 复杂度预算
- 冻结/复用的骨干：
  - 现有基于 IVF 的 BoundFetch 检索框架
  - 现有 `.clu` / 载荷存储管道
  - 现有暖基准测试工具和同步基线脚本
- 新可训练组件：无
- 新系统变更：
  - 一个 `.clu` 量化向量的暖驻聚类缓存
  - 一个原始地址块或解码就绪地址元数据的暖驻缓存
- 有意不使用的诱人添加：
  - 完整载荷预加载
  - BoundFetch 内的新索引族
  - 大调度器重新设计
  - 完整的 `search x storage` 组合基准测试

### 系统概览
1. 在索引打开或基准测试预热期间，从 `.clu` 文件预加载聚类侧量化向量。
2. 在相同的预热阶段，预加载原始地址块或解码就绪地址元数据，用于将聚类命中映射到载荷位置。
3. 将载荷字节本身保持在原始集成检索路径上。
4. 在查询执行期间，重用驻留聚类缓存，运行相同的 BoundFetch 候选过滤逻辑，仅对存活路径发起载荷侧 I/O。
5. 将结果系统测量为暖稳态 `搜索 + 载荷` 单一管道。

### 核心机制
- 输入/输出：
  - 输入：查询向量、IVF 状态、驻留 `.clu` 缓存、载荷支撑数据文件
  - 输出：Top-K 结果加上载荷访问，在一次 E2E 延迟测量中完成
- 架构或策略：
  - 驻留聚类侧元数据
  - 仅对存活候选的集成载荷获取
  - 不改变暖服务协议
- 两个实现级别：
  - `preload_codes`：仅缓存量化向量
  - `preload_codes_and_addr`：缓存量化向量加原始地址块或解码地址元数据
- 为什么这是主要创新：
  - 这不是一个通用缓存故事
  - 它具体重新平衡了在集成向量+载荷系统中什么必须留在热路径上

### 基线哲学
- 即使 DiskANN 当前获胜，也保留 DiskANN 在论文中作为强服务上界基线。
- 不要让 FlatStor 成为最终对比故事中唯一的后端。
- 使用分层基线结构：
  - 主搜索核心对比：BoundFetch vs IVF+PQ/RQ vs DiskANN，使用一个选定服务后端或一个明确记录的后端策略
  - 存储后端对比：对一到两个代表性搜索核心比较 `FlatStor`、`Lance` 和可选 `Parquet`
  - 构建/索引对比：将构建时间、索引大小和驻留预加载大小与查询延迟分开比较

### 下游管道集成
- BoundFetch 应仍被定位为服务路径系统，而非独立 ANN 核心。
- 暖驻缓存应在启动或基准测试预热时加载，然后在查询间重用。
- 论文应明确报告预加载内存占用和预加载时间，以保持优化的诚实性。

### 执行计划
- 首先实现聚类侧预加载；不要从新一轮低级 `io_uring` 调优开始。
- 在预加载前后测量延迟分解，以验证移除的工作确实在关键路径上。
- 只有在预加载结果已知后才决定是否值得进一步 BoundFetch 调优。
- 将基线套件重新设计为分阶段计划：
  - 保持 DiskANN 作为必须报告的参考
  - 保持 IVF+PQ 或 IVF+RQ 作为较弱经典族
  - 将后端格式对比移入单独支撑块而非完整矩阵

### 故障模式与诊断
- 故障模式：预加载 `.clu` 元数据改变不大。
  - 通过不变的 `e2e_ms`、不变的 `uring_submit_ms` 和不变的聚类侧时间检测。
  - 缓解：停止结构性缓存工作，得出 BoundFetch 当前受搜索质量而非热路径元数据访问限制的结论。
- 故障模式：预加载改善延迟但内存成本太大。
  - 通过驻留集大小和启动预加载时间检测。
  - 缓解：降级到部分缓存模式或按需聚类缓存，并明确报告内存权衡。
- 故障模式：评审人将后端选择解读为 cherry-picking。
  - 通过无法解释为什么 DiskANN 使用一个后端而 BoundFetch 使用另一个来检测。
  - 缓解：要么在对齐的后端上对齐，要么保持 DiskANN+FlatStor 作为明确上界参考，并将存储后端消融分离到另一张表。
- 故障模式：实验套件爆炸成太多组合。
  - 通过不断增长的 `search x storage` 矩阵且没有清晰主表来检测。
  - 缓解：冻结两阶段基线策略，并将任何额外配对视为仅附录。

### 创新性与简洁性论据
最强的更新论文故事不再是"BoundFetch 已经击败所有强暖基线"。更强且更可辩护的版本是：

"在暖集成服务中，剩余的相对于图搜索的差距来自于什么聚类侧状态留在热路径上。通过仅预加载压缩聚类表示和地址元数据，BoundFetch 可以更接近图前沿，同时不放弃其更简单的构建路径或集成载荷设计。"

这仍然是一个聚焦的系统贡献。它保持方法小，保留生产风格暖服务锚点，并保持对更强图基线存在的诚实。

## 声明驱动验证概要
### 声明 1：暖驻聚类预加载是最高价值的下一步 BoundFetch 优化。
- 最小实验：在当前最佳 BoundFetch 设置上运行 `preload=off`、`preload_codes` 和 `preload_codes_and_addr`。
- 基线/消融：
  - 当前 BoundFetch 最佳点
  - 两种预加载模式
- 指标：
  - `recall@10`
  - `e2e_ms`
  - `p99_ms`
  - `uring_submit_ms`
  - 驻留内存
  - 预加载时间
- 预期证据：
  - 在不改变召回率的情况下减少 E2E 延迟和聚类侧开销
  - BoundFetch 帕累托前沿向 DiskANN 有意义的移动

### 声明 2：DiskANN 应保留在论文中，但基线故事必须是分层的。
- 最小实验：一张主搜索核心对比和一张存储后端消融，而非完整笛卡尔积。
- 基线/消融：
  - BoundFetch + 选定后端
  - IVF+PQ 或 IVF+RQ + 相同后端
  - 如果可行 DiskANN + 相同后端，否则 DiskANN+FlatStor 作为上界参考
  - 对 `FlatStor`、`Lance` 和可选 `Parquet` 的一张代表性存储后端消融
- 指标：
  - `recall@10`
  - `e2e_ms`
  - 构建时间
  - 索引大小
- 预期证据：
  - 由于 DiskANN 被保留，主表对比保持可信
  - 存储格式贡献变得可单独解释

### 声明 3：构建时间和系统简单性现在是论文的回退价值。
- 最小实验：比较构建时间、构建期间峰值内存、启动预加载时间和驻留搜索侧内存。
- 基线/消融：
  - BoundFetch
  - DiskANN
  - 一个 IVF 家族基线
- 指标：
  - 构建时间
  - 峰值 RSS
  - 索引字节
  - 预加载字节
- 预期证据：
  - 即使 DiskANN 保持服务上界，BoundFetch 可能仍占据更好的构建时间/集成/存储效率点

## 实验交接输入
- 必须证明的声明：
  - 下一步有意义的 BoundFetch 改进是暖驻 `.clu` 预加载，而非更多盲目提交路径调优
  - DiskANN 必须作为强报告参考保留
  - 搜索核心和存储格式效应应在最终对比中分离
- 必须运行的消融：
  - 预加载关闭 vs 仅缓存代码 vs 缓存代码+地址元数据
  - 预加载后刷新 BoundFetch 帕累托
  - 分层基线对比
  - 构建时间和预加载占用表
- 关键数据集/指标：
  - COCO 100K 作为主要暖服务基准
  - `recall@10`、`e2e_ms`、`p99_ms`、`uring_submit_ms`、驻留 RSS、构建时间、预加载时间
- 最高风险假设：
  - 预加载可能不足以恢复有用的前沿区域
  - 完整后端对齐的 DiskANN 对比可能工程成本高昂
  - 存储格式对比可能爆炸除非明确分阶段
